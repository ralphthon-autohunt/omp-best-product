## 2. 에이전트 아키텍처

### 2.1 오케스트레이터 (Conductor)

전체 루프를 관리하는 메인 에이전트. 사람 역할을 대신한다.

- 아이디어 큐 관리 (생성 → 대기 → 배정)
- 빌드 파이프라인 관리 (병렬 빌드 조율)
- 결과 판정 (KILL / ITERATE / PIVOT / GRADUATE)
- 리소스 관리 (API 호출 예산, 배포 슬롯)
- 대시보드 피드 (실시간 상태 전송)

### 2.2 아이디어 에이전트 (Ideator)

**입력**: 시장 트렌드, 기존 제품 데이터, 이전 실패 교훈

**출력**:
- `product_name` — 제품명
- `one_liner` — 한 줄 설명
- `target_segment` — demographics + needs + pain_points
- `core_feature` — MVP 핵심 기능 1가지
- `differentiation` — 기존 대비 차별점
- `revenue_model` — freemium / subscription / transaction

**품질 보장**:
- 이전에 KILL된 아이디어와 유사한 건 자동 필터
- 다양성 강제: 연속 3개가 같은 카테고리면 강제 전환
- 트렌드 소스: ProductHunt, HackerNews, Reddit (사전 수집 또는 실시간)

### 2.3 빌더 에이전트 (Builder)

**입력**: Ideator의 제품 명세
**출력**: 배포 가능한 Next.js 프로젝트

**제약**:
- 템플릿 기반 (shadcn/ui + Tailwind → 일관된 품질)
- 핵심 기능 1개만 구현 (MVP의 MVP)
- 페이지 구성: 랜딩 + 핵심기능 1개 + 간단한 온보딩
- 백엔드 없음 (로컬스토리지 기반)
- 빌드 시간 5분 이내 (타임아웃 → KILL)

**빌드 전략**:
- 사전 준비된 베이스 템플릿에서 fork
- 핵심 기능만 Claude가 코드 생성
- `npm run build` 성공 필수 (실패 → 1회 재시도 → KILL)

### 2.4 디플로이어 (Deployer)

**입력**: 빌드 성공한 프로젝트
**출력**: 라이브 URL (`https://sepe-{id}.vercel.app`)

1. Vercel API로 프로젝트 생성
2. 코드 push → 빌드 → 배포
3. URL 헬스체크 (200 OK)
4. 실패 → 1회 재시도 → KILL

### 2.5 페르소나 에이전트 (Digital Twin)

> 참고: 30_research/제품실험/AI-페르소나-시뮬레이션-리서치
> 영감: MiroFish(군집지능), Stanford Generative Agents, Deepsona(합성 오디언스)

SEPE의 핵심. 제품마다 다른 페르소나 풀을 자동 생성하고, 시장 반응에 따라 증식·이탈·진화한다.

#### 페르소나 프로필 (3차원)

| 차원 | 필드 | 설명 |
|------|------|------|
| Demographics | 이름, 나이, 직업, 소득, 위치 | 세그먼트 내 상관관계 반영 (나이↔소득↔위치) |
| Behavioral | 기술 친숙도, 가격 민감도, 경쟁 제품 이력 | 제품 사용·구매 행동 결정 |
| Psychographic | Big Five 성격 (OCEAN), 리스크 태도, 혁신 수용도 | 반응 다양성의 과학적 기반 |

추가로 각 페르소나에게:
- **배경 스토리 1줄** — 판단에 영향을 주는 개인 맥락
- **네트워크** — 같은 세그먼트 내 연결된 다른 페르소나 목록
- **기억** — 이전 스프린트 경험 (ITERATE 시 태도 변화에 반영)

#### 제품별 자동 생성

Ideator가 `target_segment`를 정의하면 Persona Factory가 해당 제품에 맞는 풀을 자동 생성한다. 같은 "헬스케어 앱"이라도 "반려동물 건강"과 "운동 트래킹"은 완전히 다른 페르소나 풀이 나온다.

```
Ideator 출력 (target_segment)
    ↓
시장 컨텍스트 구성 (TAM/SAM/SOM, 경쟁 제품, 사용자 니즈)
    ↓
다차원 조합 생성 (Demographics × Behavioral × Psychographic)
    ↓
QA 필터 (비일관적 조합 제거: 예: 18세 + 연봉 $200K)
    ↓
50명 페르소나 풀 완성
```

#### 증식 (입소문 시뮬레이션)

제품이 ITERATE/GRADUATE하면 만족한 페르소나가 네트워크를 통해 새 유저를 데려온다. Bass Diffusion Model 기반.

```
만족한 페르소나 (만족도 > 0.7)
    ↓
추천 확률 = 만족도 × 영향력 × 네트워크 팩터
    ↓
네트워크 내 잠재 유저에게 추천
    ↓
잠재 유저의 채택 확률 = f(혁신 수용도, 추천자 신뢰도)
    ↓
채택 → 새 페르소나로 풀에 추가 (추천자와 유사하되 약간 다른 프로필)
```

**바이럴 계수**: K = 평균 추천 수 × 전환율
- K > 1.0 → 자연 성장 (페르소나 풀 자동 확장)
- K < 1.0 → 마케팅 필요 (Conductor가 감지)

#### 시장 확장 (Crossing the Chasm)

PMF 확보 후 시장이 커지면 페르소나 풀도 단계적으로 확장된다.

| Phase | 페르소나 | 성격 분포 | 진입 트리거 |
|-------|---------|----------|-----------|
| Innovators | 50명 | risk-taker, 불완전 OK | 초기 (기본) |
| Early Adopters | +100명 | 비전 공감, 잠재력 투자 | PMF > 40% |
| Early Majority | +200명 | 실용적, 가격 민감, 완성도 중시 | PMF > 55% + K > 0.5 |
| Late Majority | +150명 | 보수적, 업계 표준 후 진입 | 시장 점유 안정화 |

Phase 전환 시 기존 페르소나는 유지하고, 새 세그먼트의 페르소나가 추가된다.

#### 이탈 시뮬레이션

```
이탈 확률 = 기본 이탈률
    × (1 - 만족도)
    × 경쟁 제품 매력도
    × (1 - 전환 비용)
    × (1 - 충성도)
```

- 연속 3스프린트 불만족 → 이탈 + 네트워크에 부정적 입소문
- 경쟁 제품 등장 시 → 이탈 가속
- 이탈한 페르소나의 피드백은 Ideator에게 "실패 교훈"으로 전달

#### 기억과 진화 (Stanford Generative Agents 모델)

ITERATE 제품의 경우 같은 페르소나가 여러 버전을 경험한다:

1. **기억**: 이전 스프린트의 제품 경험을 자연어로 저장
2. **반성**: N 스프린트마다 상위 판단 생성 ("이 제품이 내 문제를 해결하고 있는가?")
3. **행동 변화**: 반성 결과가 다음 평가에 영향 (만족도 상승/하락, 추천 의향 변화)

이를 통해 "v1에서 실망했지만 v2에서 개선되어 만족으로 전환" 같은 현실적 행동이 나온다.

#### 비용 최적화 (하이브리드 티어)

| 티어 | 비율 | 모델 | 역할 |
|------|------|------|------|
| Deep | 10명 (20%) | Claude Sonnet | 4질문 자연어 + 구체적 피드백 + 기억/반성 |
| Mid | 15명 (30%) | Claude Haiku | Q1 판정 + Q3/Q4 키워드 응답 |
| Lite | 25명 (50%) | 규칙 기반 | Q1 확률 판정 (프로필 기반 가중치) |

### 2.6 검증 에이전트 (Validator)

1. 페르소나 50명에게 제품(URL + 설명) 제시
2. 각 페르소나가 Superhuman 4질문에 응답 (티어별 깊이 다름)
3. 결과 집계 → 판정 (GRADUATE / ITERATE / PIVOT / KILL)

**3-tier 하이브리드** (2.5절과 동일):

| 티어 | 인원 | 모델 | 응답 범위 |
|------|------|------|----------|
| Deep | 10명 | Sonnet | Q1~Q4 자연어 + 구체적 피드백 |
| Mid | 15명 | Haiku | Q1 판정 + Q3/Q4 키워드 응답 |
| Lite | 25명 | 규칙 기반 | Q1 확률 판정 (프로필 기반 가중치) |

Deep 10명의 피드백이 ITERATE 개선 방향을 결정한다.

## 6. 제품 템플릿 전략

100개를 5분 안에 빌드하려면 템플릿 기반이 필수.

### 베이스 템플릿 구조

```
sepe-template/
├── app/
│   ├── layout.tsx
│   ├── page.tsx              ← 랜딩 (히어로 + CTA)
│   └── feature/
│       └── page.tsx          ← 핵심 기능 1개 (Builder가 생성)
├── components/
│   ├── hero.tsx
│   ├── feature-card.tsx
│   └── onboarding.tsx
├── lib/
│   └── store.ts              ← 로컬스토리지 기반 상태
├── tailwind.config.ts
├── package.json
└── vercel.json
```

**Builder가 바꾸는 부분**:
- `page.tsx` — 제품명, 설명, CTA 텍스트
- `feature/page.tsx` — 핵심 기능 UI (이것만 새로 생성)
- `components/` — 필요한 UI 컴포넌트 추가
- 색상 테마, 아이콘

**Builder가 바꾸지 않는 부분**:
- 프로젝트 구조, 빌드 설정, 배포 설정
- 기본 레이아웃, 네비게이션
- shadcn/ui 컴포넌트 시스템

## 10. 에러 복구 전략

12시간 무인 자율 운영을 위한 3레이어 복구 체계.

### 에이전트 레벨 (개별 사이클)

- 빌드 실패 → 1회 재시도 → KILL
- 배포 실패 → 1회 재시도 → KILL
- 검증 API 실패 → 해당 페르소나 스킵. 응답 수 < 30이면 재실행
- 단일 사이클 타임아웃 → 20분 하드캡 → 강제 KILL + 다음 아이디어

### 파이프라인 레벨 (API·인프라)

- Claude API rate limit → 지수 백오프 (2초 → 4초 → 8초), 30초 대기 후 재시도
- Claude API 장애 (5분+ 무응답) → 대기 큐에 쌓고 복구 시 순차 처리
- Vercel 배포 큐잉 → 동시 배포 1건 제한, 나머지 큐 대기

### 시스템 레벨 (Conductor)

- Conductor 크래시 → pm2 자동 재시작
- 재시작 시 DB에서 마지막 상태 복구 (진행 중 사이클은 폐기, 다음부터 재개)
- 헬스체크: 5분마다 마지막 사이클 완료 시간 확인 → 10분+ 무반응이면 자동 재시작
- 전체 상태를 DB에 체크포인트 (사이클 시작/종료마다)
